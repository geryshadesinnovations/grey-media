1 Project Overview

Read this first. It explains, in plain words, what the system is for and who uses it.

What this application does

The Greyshades Media Platform is a private, password-protected library for a company's digital files — videos, images, PDFs and PowerPoint presentations. Think of it like a secure, branded version of Google Drive mixed with Netflix: staff log in, browse neatly organised categories, and view media safely. The files are never exposed by a direct web address; the system hands them out only to logged-in people who are allowed to see them, and it records who looked at what.

🟢 In one sentence
It is a secure “digital asset management” (DAM) website where a company stores its marketing/creative media and controls exactly who can view or download each item.

Business purpose

  • Protect valuable creative work. Designs, videos and presentations are the company's product. The platform stops casual copying and traces any leak back to the exact viewer (via a moving on-screen watermark).
  • Organise everything in one place. A single file can appear under many categories and themes without being duplicated on disk.
  • Control access. Different teams (Graphics vs Events) only see what they are allowed to see.
  • Audit everything. Every login, view, upload and download is logged for accountability.

Main features

🔵 Note

Benefits of the system

  • No expensive third-party software or monthly licences — it runs on ordinary, inexpensive web hosting.
  • No complicated installation tools (no Node.js build, no Composer required to run it).
  • Files stay private and traceable.
  • Works on phones, tablets and desktops.

2 Technology Stack

The “stack” is simply the list of tools used to build the app. Each one is explained below in plain terms.

🔵 What “frontend” and “backend” mean
Frontend = everything that runs in your web browser (what you see and click). Backend = the program running on the server (the computer in a data centre) that decides what to send back.
Backend — PHP 8.2+ the engine
What it does
PHP is the programming language running on the server. It checks logins, talks to the database, and builds the HTML pages that get sent to the browser.
Why chosen
It is available on virtually all cheap web hosts, needs no build step, and the team already knows it. A small custom “MVC” framework was written by hand instead of using a big framework like Laravel.
If removed
The entire application stops — there would be no server program to handle logins, uploads or pages.
Database — MySQL 8 / MariaDB 10.5+ the filing cabinet
What it does
Stores all the structured information: users, categories, the list of media files (but not the files themselves), logs, sessions, etc.
Why chosen
Free, extremely common, easy to manage with phpMyAdmin, and reliable.
If removed
The app cannot remember anything — no users, no media list, no logins. It would crash on the first page with “Database connection failed”.
Frontend — HTML5 + CSS3 + Vanilla JavaScript what you see
What it does
HTML is the page structure, CSS is the styling/colours/layout, and JavaScript adds interactivity (drag-drop upload, the video menu, the mobile hamburger menu).
Why chosen
“Vanilla” means plain JavaScript with no heavy framework (no React/Vue). This keeps the project tiny, fast, and means anyone who knows basic web skills can edit it.
If removed
Remove CSS → the site looks like a plain text document. Remove JS → uploads, video quality menu, search suggestions and mobile menus stop working, but pages still load.
Authentication — PHP Sessions (no JWT) the “who are you?” system
What it does
When you log in, the server remembers you using a “session” — a secret ID stored in a browser cookie. There are no JWT tokens for login; it is classic, server-side sessions.
Why chosen
Sessions are simple, secure, and let an admin instantly log someone out (important for a security-focused tool).
If removed
Nobody could stay logged in; every page would treat you as a stranger.
Storage — Server file system (private folders) where files live
What it does
Actual uploaded files live in a private storage/ folder on the server — never in the public web folder. PHP reads them and streams them to authorised users.
Why chosen
Keeps files off the public internet so their real address can never be shared.
If removed
There would be nowhere to keep uploaded media; uploads would fail.
Media tools — FFmpeg, ImageMagick, LibreOffice optional helpers
What they do
FFmpeg makes video thumbnails and multi-quality streaming versions. ImageMagick makes image/PDF thumbnails. LibreOffice converts PowerPoint into PDF for in-browser viewing.
Why chosen
Industry-standard, free command-line tools.
If removed
Uploads still succeed, but automatic previews / video quality options / PowerPoint-to-PDF are skipped. The app is built to “degrade gracefully” — it never crashes just because a tool is missing.
Libraries via CDN — hls.js, PPTXjs, Mermaid loaded from the internet

A “CDN” is a fast public file server. The app pulls a few helper libraries from cdn.jsdelivr.net:

LibraryUsed forIf removed
hls.jsPlaying multi-quality streaming video in the browserVideo falls back to a single MP4; the quality selector disappears.
PPTXjsShowing PowerPoint slides in the browser when no PDF preview existsPPT files show a “can't display” message + download option.
MermaidOnly used in this documentation page to draw diagramsDiagrams in this doc won't render; the app itself is unaffected.
Web server — Apache (with .htaccess) or Nginx the front door
What it does
The web server receives every browser request and forwards it to PHP. The .htaccess files route all traffic into one entry file and block access to private folders.
Why chosen
Apache is the default on most shared hosting and supports .htaccess rules out of the box.
If removed/misconfigured
URLs would break (404 errors) or private folders like storage/ could become publicly downloadable — a serious security risk.
🟠 No build tools
Unlike many modern websites, this project has no compile/build step. You edit a PHP/CSS/JS file and refresh — the change is live. There is no npm install or composer install needed just to run it.

🔴 Critical security rule
The web server's “document root” must point at public/ (or the root .htaccess must forward into it). If it ever points at the project root without that rule, the private storage/, app/ and .env could be downloaded by anyone. This is the single most important configuration to get right.

4 Complete File-by-File Documentation

Every file is documented below. Click a row to expand it. Each entry tells you what it does, what it depends on, and — importantly — what happens if you modify or delete it, and when you should edit it.

🔵 How to read these
Dependencies = other files this one needs. Functions = named blocks of reusable instructions inside the file. A “Controller” handles a request; a “Model” talks to the database; a “View” is an HTML template.

4.1 — Entry & configuration files

public/index.php front door
Location
public/index.php
Purpose
The single entry point. Every web request (every page) starts here. It defines folder paths, loads the app, builds the router, loads routes, and dispatches.
Dependencies
app/bootstrap.php, app/routes.php, app/Core/Router.php
Functions
None (just top-level setup code).

If you modify it: Rarely needed. Mistakes here break every page at once.

If you delete it: The website returns nothing/500 errors — there is no entry point.

When to edit: Almost never. Only to change very low-level startup behaviour.

app/bootstrap.php startup
Location
app/bootstrap.php
Purpose
Prepares the app on every request: loads config, sets the timezone, registers the autoloader (so classes load automatically), sets up error handling and logging, starts a secure session, and sends security headers (including the Content-Security-Policy that allows the jsDelivr CDN).
Dependencies
config/config.php, app/Core/helpers.php, ActivityLog
Key parts
Autoloader · session cookie settings · Content-Security-Policy header · global exception handler.

If you modify it: You can change session length, error display, or which external domains are allowed (CSP). A wrong CSP can block CDN scripts (video/PPT/diagrams).

If you delete it: The whole app fails to start.

When to edit: To allow a new CDN, change session timeout, or adjust security headers.

app/routes.php address book
Location
app/routes.php
Purpose
Maps every URL to the code that handles it, and lists which “security gates” (middleware) run first. Example: GET /dashboard → DashboardController::index behind RequireAuth.
Dependencies
All Controllers, the Middleware classes, the $router object.

If you modify it: You add/rename/remove pages and API endpoints.

If you delete it: No URLs work — the app shows only 404s.

When to edit: Whenever you add a new page or endpoint. This is one of the most commonly edited files for new features.

config/config.php settings
Location
config/config.php
Purpose
Reads the .env file and returns one big settings array (database details, storage path, upload limit, media tool paths, session/token lengths, allowed file types, video quality “renditions”, section names).
Dependencies
.env
Functions
env() — read one setting with a default.

If you modify it: Change allowed file types, upload size, video qualities, or defaults.

If you delete it: The app can't read any settings → fails to start.

When to edit: To allow a new file type, add a video quality, or change a default.

.env secrets
Location
project root .env
Purpose
Holds environment-specific secrets and paths: database host/name/user/password, app URL, storage path, tool paths, feature flags.
🔴 Never share or commit
This file contains the database password. It must never be public or stored in Git.

If you modify it: Point at a different database, change the site URL, toggle watermark/anti-inspect.

If you delete it: The app falls back to defaults and almost certainly can't reach the database.

When to edit: When deploying to a new server, or rotating the DB password.

🟠 Known gotcha
Make sure DB_PORT is the database port (usually 3306), not 80.
.htaccess (root) & public/.htaccess routing + security
Location
root .htaccess and public/.htaccess
Purpose
Apache rules. The root one forwards all traffic into public/. The public one sends every request to index.php (unless a real file exists) and adds security headers; Options -Indexes stops folder listings.

If you modify it: Change routing or headers (Apache only).

If you delete it: URLs break and/or private folders may become browsable. Nginx hosts use server config instead (see README).

When to edit: Rarely — only for server-level routing/header tweaks.

4.2 — Core building blocks (app/Core/)

app/Core/Database.php DB connection
Purpose
A single, shared connection to the database using PDO (PHP's safe database layer). Provides helpers that use “prepared statements” to prevent hacking via SQL injection.
Functions
pdo() connect · all() many rows · first() one row · scalar() one value · execute() insert/update/delete · lastId() new row's ID.

If you modify it: Change connection options. Risky — touches all data access.

If you delete it: Nothing can read/write data; app crashes.

When to edit: Almost never. Use its helpers from Models instead.

app/Core/Auth.php login & permissions
Purpose
Everything about “who is logged in and what may they do”. Verifies passwords, regenerates the session for safety, records logins, and answers permission questions.
Functions
attempt() log in · logout() · check() logged in? · user()/id() · isSuperAdmin() · flag() read a permission switch · canSection() · allowedSections() · canAccessSections() · canUpload()/canEdit()/canDelete()/canDownload()/canManageUsers().

If you modify it: Change how login or permissions work for everyone.

If you delete it: No logins, no permission checks — total breakage.

When to edit: To add a new permission rule or change section-access logic. See Section 8.

app/Core/Router.php URL matcher
Purpose
Matches the requested URL against the routes, runs the listed middleware, and calls the right controller method. Supports parameters like /media/{uuid} and a _method override for forms.
Functions
get/post/put/delete/any() register routes · dispatch() run · call() invoke handler.

If you modify it: Change how routing works globally.

If you delete it: No request can reach any controller.

When to edit: Rarely — only to change routing behaviour itself (not to add routes; do that in routes.php).

app/Core/View.php template engine
Purpose
Renders an HTML template file and supports “layouts” (a page frame) with named sections (e.g. extra scripts).
Functions
render() · extend() · section()/endSection().

If you modify it: Change how all pages are assembled.

If you delete it: No HTML page can be produced.

When to edit: Rarely — only to add template features.

app/Core/Csrf.php form safety
Purpose
Protects against “CSRF” attacks (a trick where another site submits a form as you). Generates a secret token, prints it as a hidden field, and verifies it on every form submit.
Functions
token() · field() hidden input · check() · verifyOrFail() (returns 419 on mismatch).

If you modify it: Change token handling. Risky.

If you delete it: Forms become vulnerable; many controllers call it and would error.

When to edit: Rarely.

app/Core/StreamToken.php video/file passes
Purpose
Issues short-lived (15-minute) “tickets” that allow a specific user to stream a specific file. Stored in the database so logging out instantly cancels them.
Functions
issue() · validate() · revokeForUser() · purgeExpired().

If you modify it: Change how long streaming links last or how they're checked.

If you delete it: Secure streaming breaks; videos/files won't load.

When to edit: To change token lifetime or add encryption later.

app/Core/ActivityLog.php audit trail
Purpose
Writes a record of meaningful actions (login, view, upload, edit, delete, download) with user, IP, browser and session. Failures are swallowed so logging never breaks a page.
Functions
record() · error().

If you modify it: Change what/how things are logged.

If you delete it: Many controllers call it — they'd error; the audit trail disappears.

When to edit: To log a new kind of action.

app/Core/helpers.php small utilities
Purpose
Tiny global helper functions used throughout the app.
Functions
config() read a setting · e() escape HTML (safety) · url() · asset() (adds ?v= cache-busting) · redirect() · old() · flash() one-time messages · storage_path() · slugify() · client_ip() · ua() · format_bytes() · format_duration() · view().

If you modify it: Affects formatting/links everywhere.

If you delete it: The app crashes (these are used constantly).

When to edit: To add a new small helper used across pages.

AdminController.php admin panel

Functions

When to edit: Change thumbnail sizes, video qualities, or how PPT/PDF previews are made.

8 Authentication & Authorization

Authentication = “proving who you are” (login). Authorization = “what you're allowed to do” (permissions).

Login flow (step by step)

sequenceDiagram participant U as User participant A as AuthController.login() participant DB as Database U->>A: submit email + password A->>A: CSRF check + email format A->>DB: count recent failed_logins for this IP alt 5+ fails in 15 min A-->>U: "Too many attempts" (blocked) else allowed A->>DB: find active user by email A->>A: password_verify(password, stored hash) alt wrong A->>DB: insert failed_logins row A-->>U: "Invalid credentials" else correct A->>A: session_regenerate_id() (anti-fixation) A->>DB: update last_login + user_sessions A->>DB: activity_logs "login" A-->>U: redirect to /dashboard end end

Session flow (staying logged in)

  • On success, the server stores your user details in a server-side session and sets a cookie named GREYSHADES_SID in your browser.
  • The cookie is HttpOnly (JavaScript can't read it), SameSite=Lax, and Secure over HTTPS — these make it hard to steal.
  • On each protected page, RequireAuth confirms the session exists and refreshes your “last active” time.
  • Logging out empties the session, deletes the cookie, and cancels all your streaming tickets.
🔵 No JWT
This app deliberately uses classic server sessions, not JWT tokens, so an admin can instantly revoke access.

Roles & the permission model

Two layers decide access: the role (a floor) and per-user switches (fine control). Super Admin overrides everything.

RoleSees GraphicsSees EventsDefault extra powers
super_admin✅✅All (upload/edit/delete/download/manage users)
graphics_user✅ (if can_graphics)—View only by default
events_user—✅ (if can_events)View only by default
combined_user✅✅View only by default

Exactly how a permission is checked

In code, checks read the logged-in user's switches. Examples from Auth.php:

// Can this user use a section? Super admin always can.
canSection('events')  →  isSuperAdmin()  ||  user['can_events'] == 1

// Can they upload?
canUpload()  →  isSuperAdmin()  ||  user['can_upload'] == 1

// Can they SEE a specific media item?
// They can if ANY section the item touches is allowed to them.
canAccessSections( Media::sectionCodesFor(item) )
🟢 Why sectionCodesFor matters
A file's section is decided by its categories. Because a file can be filed under both Graphics and Events categories, visibility is based on every section it touches — so an Events user can see a file that is (also) in an Events category, even if its “primary” section is Graphics. This was a deliberate fix.

13 Category System

Categories are a tree: a few top-level “parents”, each with “children”, which can have their own children.

Parents & children

  • Graphics section parents: Gimmick, Art, Hybrid. Example chain: Art → Videos → Invitation Videos.
  • Events section parent: Events (Conference activities, Stall fabrication, …).
  • A category points to its parent via parent_id. Top-level categories have no parent.

“Attach all parents” rule (important)

When you upload/edit and tick a leaf category (e.g. “Invitation Videos”), the backend automatically also attaches its parents (“Videos”, “Art”). This is done by Category::ancestorIds() in UploadController/MediaController. That's why filtering by either the leaf or the top-level returns the file.

Where category COUNTS are calculated

🟢 Exact answer
The little “(2)” count badge on each category card is calculated in public/assets/js/upload.js, in the refreshCounts() function. It counts the checked checkboxes inside each .cat-card and writes the number into .cat-card-count. It runs on both the Upload and Edit pages.

Search-box suggestion counts (how many files are under a category/company) are calculated server-side in Media::suggest() using COUNT() queries.

Category filters on the dashboard

Clicking a category in the sidebar reloads /dashboard?category={id}. Media::search() then matches that category and all of its descendants (via Category::descendantIds()), so picking a parent shows everything beneath it.

15 Change Request Guide

Common “I want to…” requests, with the exact files to edit and what to test afterwards.

“I want to add a new page”
  1. Add a route in app/routes.php (e.g. $router->get('/reports', [ReportController::class,'index'], [RequireAuth::class]);).
  2. Create the controller in app/Controllers/ with an index() method that calls view('reports/index', [...]).
  3. Create the view in app/Views/reports/index.php (start with $this->extend('layouts/app');).
  4. (Optional) add a link in layouts/app.php or the relevant nav.

Test: visit the URL logged in and logged out; confirm the gate works.

“I want to add a new role”
  1. Add the role to the roles table (and seed.sql for fresh installs).
  2. Decide its section access & powers; reflect them in the per-user can_* switches when creating users.
  3. If the role needs special logic, update Auth.php (e.g. a new canX() or allowedSections() rule).
  4. Show the role in admin/users.php dropdown (it reads User::roles() automatically).

Test: create a user with the new role; verify what they can/can't see.

“I want to add a new category”

Easiest: Admin → Categories → choose section + parent + name. Done.

For fresh installs: add an INSERT in seed.sql.

Test: it appears in the dashboard sidebar and in the upload/edit category picker.

“I want to change the dashboard design”
  • Layout/markup → Views/dashboard/index.php and _card.php.
  • Colours/spacing/grid → app.css (look for .media-grid, .media-card, .filterbar).
  • Behaviour (hover preview) → app.js.

Test: desktop + mobile; check filters and pagination still work.

“I want to modify upload functionality”
  • UI/validation messages → upload/form.php + upload.js.
  • Server validation, dedupe, storage → UploadController::store().
  • Allowed types/limits → config.php.
  • Preview generation → MediaProcessor.php.

Test: each file type; a duplicate; an oversize file; a missing-thumbnail case.

“I want to add a new media type (e.g. audio)”
  1. Add its MIME to config.php allowed_mimes.
  2. Add classification in MediaProcessor::classify() and the media_type enum in the DB.
  3. Add a viewer branch in media/show.php.
  4. Add an extension mapping in UploadController::extFor().

Test: upload + view + thumbnail + download.

“I want to log a new action”

Call ActivityLog::record('media.share', 'media', $id, ['to'=>$email]); from the relevant controller. It appears in Admin → Activity.

16 Developer Onboarding Guide

Step-by-step setup for a brand-new developer or server.

  1. Install PHP 8.2+ and MySQL/MariaDB. (Optional: FFmpeg, ImageMagick, LibreOffice for previews.)
  2. Get the code:

17 Security Documentation

Security is layered (“defence in depth”). No single layer is perfect, but together they make misuse hard and traceable.

Authentication security

  • Passwords stored as bcrypt hashes, never plain text; re-hashed if the algorithm strengthens.
  • Session fixation prevented by regenerating the session ID on login.
  • Login throttling: 5 failures per IP per 15 minutes.
  • Cookies are HttpOnly, SameSite=Lax, and Secure on HTTPS.

Authorization security

  • Every protected route runs RequireAuth/RequireAdmin before the controller.
  • Section visibility is enforced in queries and in the viewer/stream authorisation.
  • Downloads are separately gated (is_downloadable + per-user grants) and fully logged.

File-upload security

  • File type is detected from content (finfo), not just the name, and checked against an allow-list.
  • Files are stored with random UUID names in a private folder, served only through PHP with a valid token.
  • Real file paths never appear in any URL.

Database security

  • All queries use prepared statements (parameters sent separately) — this blocks SQL injection.
  • The DB user should have only the privileges it needs.

API / output security

  • Output is escaped with e() to prevent XSS (malicious scripts in data).
  • A strict Content-Security-Policy limits where scripts/styles/media can come from.
  • Headers: X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy.
  • Path-traversal guards on streaming (basename() + real-path checks).
🟠 Honest limitation
The browser “anti-copy” deterrents (security.js) and the watermark raise the effort to leak content and make leaks traceable, but cannot guarantee 100% prevention (a determined user can screen-record). The real protections are server-side: token-gated streaming, audit logs, and the watermark.

20 Future Improvements

Honest recommendations to make the platform faster, safer and easier to scale.

Recommended improvements

  • Background job queue for video transcoding (don't do it during the upload request) — see technical debt below.
  • Forgot/Reset password flow and optional two-factor login.
  • Live permission refresh so changes apply without re-login.
  • Cross-browser PDF viewer (PDF.js) to hide the download button consistently, including Firefox.
  • Bulk actions (multi-select cards to tag/categorise/delete).
  • Per-file download grants UI in the admin panel.
  • Server tools health page showing whether FFmpeg/ImageMagick/LibreOffice are installed.
